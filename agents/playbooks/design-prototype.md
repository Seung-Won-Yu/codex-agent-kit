# Design And Prototype Quality Checklist

Use only for substantial or ambiguous product/design work, visual direction, prototypes, or “something like this.” Preserve the request's intent and effect ceiling; recommendations do not authorize creation or edits.

- Identify the real deliverable: critique, direction, visual concept, working prototype, or production implementation.
- When two paths would materially change cost or output, show only `내 해석`, `추천 경로`, and the meaningful alternative.
- Design around the user's workflow and useful states, not a component list or generic marketing page.
- Reuse the project's design system and interaction patterns when they exist.
- Include loading, empty, error, disabled, and confirmation states only when the experience needs them.
- Check responsive stability, accessibility, text overflow, feedback, and important actions.
- For visible implementation, use browser-based verification when feasible.
- Keep visual exploration separate from workspace changes unless the user authorized creation or modification.
